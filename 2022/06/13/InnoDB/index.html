00-基本概念

持久化（persistence）：把数据保存到可掉电式存储设备中以供之后使用。大多数情况下，特别是企业级应用，数据持久化意味着将内存中的数据保存到硬盘上加以”固化”，而持久化的实现过程大多通过各种关系数据库来完成

数据库（DB-Database）:即存储数据的“仓库”，其本质是一个文件系统。它保存了一系列有组织的数据

数据库管理系统（DBMS-Database Management System）:是一种操纵和管理数据库的大型软件，用于建立、使用和维护数据库，对数据库进行统一管理和控 制。用户通过数据库管理系统访问数据库中表内的数据

结构化查询语言（ SQL-Structured Query Language）:专门用来与数据库通信的语言

关系型数据库（RDBMS） ：

  • 定义：把复杂的数据结构归结为简单的二元关系 （即二维表格形式），以 行(row) 和 列(column) 的形式存储数据，以便于用户理解。
  • 优点：
    • 复杂查询：可以用SQL语句方便的在一个表以及多个表之间做非常复杂的数据查询
    • 事务支持：使得对于安全性能很高的数据访问要求得以实现。

非关系型数据库：

  • 定义：可看成传统关系型数据库的功能阉割版本 ，基于键值对存储数据，不需要经过SQL层的解析， 性能高
  • 类型：键值型，文档型，搜索引擎，列式，图

表、记录、字段：一个实体集（class）对应于数据库中的一个表（table），一个实体（instance）则对应于数据库表 中的一行（row），也称为一条记录（record）。一个属性（attribute）对应于数据库表中的一列 （column），也称为一个字段（field）

SQL（Structure Query Language，结构化查询语言）

  • 是使用关系模型的数据库应用语言， 与数据直接打交道 ，由 IBM 上世纪70年代开发出来。后由美国国家标准局（ANSI）开始着手制定SQL标准， 先后有 SQL-86 ， SQL-89 ， SQL-92 ， SQL-99 等标准

  • SQL语言按功能分类：

    • DDL（Data Definition Languages 数据定义语言），这些语句定义了不同的数据库、表、视图、索引等数据库对象，还可以用来创建、删除、修改数据库和数据表的结构。 主要的语句关键字包括 CREATE 、 DROP 、 ALTER 等
    • **DML**（Data Manipulation Language 数据操作语言），用于添加、删除、更新和查询数据库记录，并检查数据完整性。 主要的语句关键字包括 INSERT、DELETE、UPDATE、SELECT等
    • DCL（Data Control Language 数据控制语言），用于定义数据库、表、字段、用户的访问权限和安全级别。 主要的语句关键字包括GRANT、REVOKE、COMMIT、ROLLBACK、SAVEPOINT 等

  • 语言规范：windows对大小写不敏感，但Linux对大小写敏感

约束

数据完整性（Data Integrity）是指数据的精确性（Accuracy）和可靠性（Reliability）。它是防止数据库中存在不符合语义规定的数据和防止因错误信息的输入输出造成无效操作或错误信息而提出的。

为了保证数据的完整性，SQL规范以约束的方式对表数据进行额外的条件限制。从以下四个方面考虑：

  • 实体完整性（Entity Integrity） ：例如，同一个表中，不能存在两条完全相同无法区分的记录
  • 域完整性（Domain Integrity） ：例如：年龄范围0-120，性别范围“男/女”
  • 引用完整性（Referential Integrity） ：例如：员工所在部门，在部门表中要能找到这个部门
  • 用户自定义完整性（User-defined Integrity） ：例如：用户名唯一、密码不能为空等，本部门 经理的工资不得高于本部门职工的平均工资的5倍。

创建表时规定约束（通过 CREATE TABLE 语句），或者在表创建之后通过 ALTER TABLE 语句规定约束。

约束的分类：

  • 根据约束数据列的限制，约束可分为：
    • 单列约束：每个约束只约束一列
    • 多列约束：每个约束可约束多列数据
  • 根据约束的作用范围，约束可分为：
    • 列级约束：只能作用在一个列上，跟在列的定义后面
    • 表级约束：可以作用在多个列上，不与列一起，而是单独定义

视图

视图一方面可以帮我们使用表的一部分而不是所有的表，另一方面也可以针对不同的用户制定不同的查询视图。比如，针对一个公司的销售人员，我们只想给他看部分数据，而某些特殊的数据，比如采购的价格，则不会提供给他。再比如，人员薪酬是个敏感的字段，那么只给某个级别以上的人员开放，其他 人的查询视图中则不提供这个字段。

定义：

  • 视图是一种虚拟表 ，本身是 不具有数据的，占用很少的内存空间，它是 SQL 中的一个重要概念。
  • 视图建立在已有表的基础上, 视图赖以建立的这些表称为基表。
  • 视图的创建和删除只影响视图本身，不影响对应的基表。但是当对视图中的数据进行增加、删除和修改操作时，数据表中的数据会相应地发生变化，反之亦然
  • 向视图提供数据内容的语句为 SELECT 语句, 可以将视图理解为存储起来的 SELECT 语句
  • 视图，是向用户提供基表数据的另一种表现形式。通常情况下，小型项目的数据库可以不使用视图，但是在大型项目中，以及数据表比较复杂的情况下，视图的价值就凸显出来了，它可以帮助我们把经常查询的结果集放到虚拟表中，提升使用效率，理解和使用起来都非常方便。

优点：

  • 操作简单
  • 减少数据冗余
  • 数据安全：用户不需要查询数据表，可以直接 通过视图获取数据表中的信息，具有隔离性
  • 适应灵活多变的需求
  • 能够分解复杂的查询逻辑

缺点

  • 如果实际数据表的结构变更了，我们就需要及时对相关的视图进行相应的维护。特别是嵌套的视图（就是在视图的基础上创建视图），维护会变得比较复杂， 可读性不好 ，容易变成系统的潜在隐患。

存储过程

含义：存储过程的英文是 Stored Procedure 。它的思想很简单，就是一组经过预先编译 的 SQL 语句的封装。

执行过程：存储过程预先存储在 MySQL 服务器上，需要执行的时候，客户端只需要向服务器端发出调用存储过程的命令，服务器端就可以把预先存储好的这一系列 SQL 语句全部执行。

好处：

  1. 简化操作，提高了sql语句的重用性，减少了开发程序员的压力
  2. 减少操作过程中的失误，提高效率
  3. 减少网络传输量（客户端不需要把所有的 SQL 语句通过网络发给服务器）
  4. 减少了 SQL 语句暴露在网上的风险，也提高了数据查询的安全性

01-MySQL体系结构

定义数据库和实例

  • 数据库：物理操作系统文件或其他文件类型的集合，是依照某种数据模型组织起来并存放与二级存储器的数据集合
  • 实例：是操作数据库文件的程序，是位于用户与操作系统之间的一层数据管理软件
  • 关系：
    • 实例与数据库是一一对应的关系（集群情况下存在一个数据库被多个实例引用的情况）
    • 实例在系统上的表现就像一个进程
  • 启动实例：MySQL数据库会去读取配置文件的参数（如果有多个配置文件中有同一个参数，以最后读取的为准）

MySQL体系结构：

MySQL区别于其他数据库的最主要特点就是其插件式存储引擎（存储引擎是基于表的，而不是基于数据库的）

存储引擎

  • 好处：开发人员可根据开发需求自行调整存储引擎的结构
  • 种类：
    • InnoDB
      • 适用：面向在线事务处理（OLTP）的应用
      • 特点：行锁设计、支持外键、支持非锁定读、多版本并发控制（MVCC）、聚集存储
    • MyISAM（5.5.8以前默认）
      • 适用：OLAP数据库应用
      • 特点：不支持事务、表锁设计、支持全文索引、只缓存索引文件
    • NDB
      • 集群存储引擎
      • 特点：数据全部放在内存、主键查找速度快、增加数据存储节点可先行提高性能、网络开销大
    • Memory
      • 适用：存储临时数据的临时表、数据仓库的纬度表
      • 特点：使用哈希索引、速度非常快、只支持表锁、并发性能差、不支持BLOB和TEXT列类型（遇到这些字段MySQL自动转换到MyISAM放到磁盘）、浪费内存（存储变长字段时按常字段方式进行的）、
    • Archive
      • 适用：存储归档数据
      • 特点：只支持INSERT和SELECT操作（5.1开始支持索引）、高速的插入和压缩、并不是事务安全的、
    • Federated
      • 不存放数据，只是指向一台远程MySQL数据库服务器上的表
    • Maria
      • MyISAM升级版
      • 特点：支持缓存数据和索引文件、行锁设计、多版本并发控制、支持事务和非事务安全的选项、优化BLOB字符类型的处理性能
    • 其他

连接MySQL

连接MySQL操作是一个连接进程和MySQL数据库实例进行通信，实质上就是两个进程间的通信

  • TCP/IP
    • 使用最广，Mysql在任何平台下都提供的连接方式
    • 一般情况下客户端和数据库实例在不同的服务器上，通过TCP/IP网络连接
    • 在连接时，MySQL数据库会根据权限视图先检查客户端的是否允许连接到数据库
  • 命名管道和共享内存
    • Windows平台
    • 两个需要通信的进程在同一台服务器上可以使用命名管道（–enable-named-piped）
    • 4.1之后提供共享内存的连接方式（–shared-memory，同时客户端–protocol=memory）
  • UNIX域套接字
    • Linux和UNIX环境下
    • 并非网络协议，所以只能在一台服务器的进程通信上使用

02-InnoDB存储引擎

InnoDB体系架构

后台线程

  • Master Thread（核心）：将缓冲池中的数据异步刷新到磁盘，保证数据一致性。脏页刷新、合并插入缓存、undo页回收等
  • IO Thread：负责IO请求的回调处理
  • Purge Thread：回收已经被分配并使用的undo页
  • Page Cleaner Thread：新版本加入，将脏页刷新操作放到新的线程里面，减轻master thread的工作和查询线程的阻塞

内存

缓冲池
  • 通过内存的速度来弥补磁盘速度较慢对数据库性能的影响
  • 缓冲池的大小直接影响数据库的整体性能（缓冲池大小参数：innodb_buffer_pool_size）
  • 允许有多个缓冲池实例，每个页根据哈希值分配到不同的缓冲池实例中，减少数据库内部的资源竞争，增加并发能力（缓冲池数量参数：innodb_buffer_pool_instances，默认为1）
  • SHOW ENGINE INNODB STATUS可以观察每个缓冲池实例对象的运行状态，5.6后也可以通过information_schema架构下的BUFFER_POOL_STATS来观察缓冲状态

LRU LIST

  • 数据库中的缓冲池是通过LRU（Latest Recent Used）算法进行管理的
  • 最频繁使用的页在LRU列表的前端，最少使用的在尾端，当缓存池满释放尾端的页
  • midpoint：
    • 新访问的页放在midpoint位置（参数：innodb_old_blocks_pct，默认5/8位置）
    • midpoint之前的列表称为new列表（活跃的热点数据），之后的列表称为old列表
    • 为什么不把新读取的页放在列表首部呢？可能会导致需要的热点数据被移除，影响缓冲池的效率
    • innodb_old_blocks_time：表示页读取到mid位置后需要等待多久才会被加入到LRU的热端，适当延长等待时间可以尽可能保证热点数据不被刷出
  • 运行过程：
    • 数据库启动之初，LRU列表为空，所有的页都存放在Free列表中
    • 当需要从缓冲池分页时，先看Free List中是否有空闲页，若有则将该页从Free List删除，放入LRU列表
    • 否则，淘汰LRU列表末尾的页，将该内存空间分配给新的页
    • 当页从LRU LIST的old部分加入到new部分时，发生的操作叫page made young，页处在等待时间尚未加入new部分的操作叫page not made young
  • 压缩页：
    • InnoDB存储引擎支持压缩页的功能，将原本16kb的页压缩到1kb,2kb,4kb,8kb，对应非16kb的页，通过unzip_LRU列表管理（包含在LRU列表中）
    • 不同大小压缩页的内存分配（4kb为例）：
      1. 检查4kb的unzip_LRU列表，检查是否有可用的空闲页
      2. 若有则直接使用
      3. 否则，检查8kb的unzip_LRU列表
      4. 若能得到空闲页，将页分成两个4kb页放到4kb的unzip_LRU列表
      5. 若不能，从LRU列表申请一个16kb的页，分成一个8kb，两个4kb的页，分别存放到对应unzip_LRU列表
  • 脏页：
    • LRU列表中的页被修改后，该页称之为脏页（即缓冲池中的页和磁盘上的页的数据产生了不一致）
    • Flush列表即为脏页列表，通过CHECKPOINT机制将脏页刷新回磁盘
    • 注意：脏页既存在于LRU列表（管理缓冲池中页的可用性），也存在于Flush列表（管理页刷新回磁盘），二者互不干扰
重做日志缓冲
  • InnoDB存储引擎首先将重做日志信息放入这个缓冲区，然后按一定频率将其刷新到重做日志文件
  • 一般每一秒都会刷新日志文件，因此重做日志缓冲区不必特别大（保证大于每秒事务量即可，通常设置为8mb），控制参数：innodb_log_buffer_size
  • 刷新触发情况：
    • Master Thread每秒
    • 每个事务提交时
    • 当重做日志缓冲池剩余空间小于1/2时
额外内存池
  • 在InnoDB存储引擎中，对内存的管理是通过一种称为内存堆（heap）的方式进行的
  • 在对一些数据结构本身的内存进行分配时，需要从额外的内存池中进行申请，当该区域的内存不够时，会从缓冲池中进行申请。
  • 例如，分配了缓冲池（innodb_buffer_pool），但是每个缓冲池中的帧缓冲（framebuffer）还有对应的缓冲控制对象（buffer control block），这些对象记录了一些诸如LRU、锁、等待等信息，而这个对象的内存需要从额外内存池中申请。因此，在申请了很大的InnoDB缓冲池时，也应考虑相应增加额外内存池的大小

CHECKPOINT技术

前提：

  • 当前数据库普遍采用Write Ahead Log策略，即当事务提交时，先写重做日志，再修改页，当发生宕机等意外时通过日志文件完成数据的恢复
  • 缓冲池不可能缓存数据库中的索引数据
  • 重做日志不可能无限大，就算无限大也不可能从头开始按日志重新恢复，耗时太长

作用：保证checkpoint之前的页都已经刷新回磁盘，数据库就可以不用重做所有的日志

LSN（Log Sequence Number）日志序列号，占8字节，单调递增

  • 含义：页的版本、重做日志写入总量、checkpoint的位置

分类：

  • Sharp Checkpoint：数据库关闭时将所有脏页刷回磁盘
  • Fuzzy Checkpoint：只刷新一部分脏页
    • Master Thread Checkpoint：每秒或每十秒将脏页列表中异步刷新一定比例的页回磁盘
    • FLUSH_ LRU_ LIST Checkpoint：若LRU中可用空闲页不够，存储引擎会移除尾端页，如果该页是脏页就需要checkpoint（LRU可用页参数：innodb_lru_scan_depth）
    • Async/Sync Flush Checkpoint：保证重做日志的循环可用性
      • 当前事务数据库系统对重做日志的设计都是循环使用的，不再需要的日志部分（数据库恢复操作不需要这部分日志）可以被覆盖重用，若此时将被覆盖的重做日志还需要使用，出现重做日志不可用的情况，就进行强制checkpoint，将缓冲池中的页至少刷新到当前重做日志的位置（LSN：已经写入重做日志记为redo_lsn；已经刷新回磁盘记为checkpoint——lsn）
    • Dirty Page too much Checkpoint：脏页数量太多，导致InnoDn存储引擎强制checkpoint（innodb_max_dirty_page_pct：最大脏页百分比）

Master Thread工作方式

概述：

  • Master Thread具有最高的线程优先级
  • 由多个循环组成：主循环、后台循环、刷新循环、暂停循环，根据数据库运行状态在循环间切换

主循环（Loop）：

  • loop循环通过thread sleep来实现，可能会出现延迟
  • 每秒一次的操作：
    • 日志缓冲刷新到磁盘，即使这个事务还没提交（总是）
    • 合并插入缓冲（可能）
    • 刷新至多100个缓冲池中的脏页至磁盘（可能）
    • 如果当前还没有用户活动，切换到后台循环（可能）
  • 每十秒一次的操作：
    • 刷新100个脏页到磁盘（可能）
    • 合并至多5个插入缓冲（总是）
    • 将日志缓冲刷新到磁盘（总是）
    • 删除无用的undo页（总是）
    • 刷新100个或10个脏页到磁盘（总是）

**后台循环**（background loop）：

  • 若当前没有用户活动（数据库空闲）或数据库关闭时，会切换到后台循环
  • 操作：
    • 删除无用的undo页（总是）
    • 合并20个插入缓冲（总是）
    • 跳回主循环（总是）
    • 不断刷新100个页直到符合条件（可能，跳转到刷新循环执行）

刷新循环（flush loop）

暂停循环（suspend loop）：刷新循环也处于空闲，innodb存储引擎会切换到暂停循环，等待事件发生

若用户启用了innodb存储引擎，但没有使用任何innodb存储引擎的表，master Thread总是处于挂起状态

——————————1.2版本前：——————————

存在问题：在写入密集的应用程序中，master thread的刷新速度不够用，如果发生宕机，恢复时间需要很久

1.0后改进：

  • 引入参数innodb_io_capacity:

    • 在合并插入缓冲时，合并插入缓冲的数量为innodb_io_capacity的5%

    • 在从缓冲区刷新页的时候，刷新页的数量为innodb_io_capacity

  • 调整innodb_max_dirty_page_pct：过大增加内存压力，过小增加磁盘压力，1.0之后调整到75%

  • 引入参数innodb_adaptive_flushing（自适应刷新）：函数buf_flush_get_desired_flush_rate通过判断产生重做日志的速度决定最合适的刷新脏页数量

  • 引入参数innodb_purge_batch_size：控制每次full purge时回收的undo页数量，默认20

——————————1.2版本：——————————

InnoDB关键特性

插入缓存（提升性能）

——————————1.0版本前：——————————

Inser Buffer：

  • 运行过程：对于非聚集索引的插入或更新操作，不是每一次都直接插入到索引页当中，而是先判断插入的非聚集索引页是否在缓冲池中，若在，则直接插入；若不在，则先放入一个insert buffer对象中，insert buffer和辅助索引子节点按一定频率进行合并，
  • 好处：就能将多个插入合并到一个操作当中，提高对非聚集索引插入的性能
  • 缺点：写密集的操作下，插入缓存会占用大量的缓冲池内存（最大占用1/2，参数inuf_pool_size_per_max_size）
  • 适用对象：非唯一的辅助索引

——————————1.0版本后：——————————

Change Buffer:

  • 升级版Insert Buffer，可对DML（insert、delete、update）操作都进行缓冲（insert buffer、delete buffer、purge buffer）
  • 适用对象：非唯一的辅助索引
  • 运行过程：
    • 将记录标记为已删除（对应delete buffer）
    • 真正将记录删除（对应purge buffer）
  • 参数：
    • innodb_change_buffering：可选inserts、deletes、purges、changes、all、none
    • 1.2版本引入innodb_change_buffer_max_size（默认25，最大50，百分比）

————————————————————————

内部实现

  • 数据结构：B+树，现版本全局只有一棵Insert Buffer B+树

  • 插入步骤：当一个辅助索引要插入到页时，如果这个页不在缓冲池当中时，那么InnoDB存储引擎首先创造一个search key（非叶子节点），接下来查询B+树，将这条记录插入到叶子节点

  • 非叶子节点存放查询的键值（search key）：

    • space：4字节，有唯一的space id，查询得知所属表
    • marker：1字节，兼容旧版本插入缓存
    • offset：4字节，页所在的偏移量

  • 叶子节点：

    • 前面与非叶子节点结构相同

    • metadata：

      名称 字节 作用
      IBUF_REC_OFFSET_COUNT 2 两字节整数，排序每个记录进入insert buffer的顺序
      IBUF_REC_OFFSET_TYPE 1
      IBUF_REC_OFFSET_FLAGS 1

Merge Insert Buffer

  • 插入缓冲中的记录何时合并到真正的辅助索引：

    • 辅助索引页被读取到缓冲池：例如此时执行select语句，需要检查bitmap页，看是否有记录存放在B+树中，若有，则将该记录插入到辅助索引页中
    • Insert Buffer Bitmap页追踪到该辅助索引页已无可用空间：若插入辅助索引记录时检测到插入记录后可用空间会小于1/32，会强制进行一次合并
    • Master Thread：主线程每秒或每十秒都会有合并操作

  • Insert Buffer Bitmap：

    • 作用：用来追踪辅助索引页

    • 结构：每个辅助索引页在Insert Buffer Bitmap里占用4位

      名称 大小 说明
      IBUF_BITMAP_FREE 2 表示该辅助索引页中的可用空间数量
      IBUF_BITMAP_BUFFERED 1 1表示该辅助索引有记录被保存在B+树中
      IBUF_BITMAP_IBUF 1 1表示该页为B+树的索引页

两次写（可靠性）

前提：

  • 当数据库发生宕机时，Innodb存储引擎正在写入某个页到表中，写操作只完成了一部份，这种情况称为部分写失效。
  • 重做日志记录的是对页的物理操作，在页损坏的情况下重做没有意义

解决方法：在应用重做日志前，用户需要一个页的副本，当发生写入失效的时候，先通过页的副本来还原该页，再进行重做

运行步骤：

  • 写入
    1. 对缓冲池中的脏页进行刷新时，并不直接写磁盘，而是通过memory函数将脏页先复制到内存的doublewrite buffer
    2. doublewrite buffer分两次（每次1mb）顺序的写入共享表空间的物理磁盘上
    3. 马上调用同步函数同步磁盘
  • 恢复
    1. Innodb存储引擎可以从共享表空间的doublewrite中找到该页的一个副本
    2. 将副本复制到表空间文件
    3. 应用重做日志

参数：

  • Innodb_dblwr_pages_written和innodb_buffer_pool_pages_flushed：生产环境中的写入量和缓冲池中刷新到磁盘页的数量（所有页的刷新都要先放入doublewrite中，而这理论上应该相等，但后者5.5.24以前有bug）
  • skip_innodb_doublewrite：禁用doublewrite功能

自适应哈希索引（AHI）

工作原理：innodb存储引擎会监控对表上个索引页的查询，如果观察到建立哈希索引可以带来速度的提升，则建立哈希索引，称为自适应哈希索引（AHI）

要求：

  • 对这个页的连续访问模式必须一样
  • 以该模式访问了100次
  • 页通过该模式访问了N次（N = 页中记录 / 16）

优化：读取写入速度提高2倍，辅助索引的连接操作性能提高5倍

注意：哈希索引只能用来搜索等值的查询，不适用于其它类型的查找

异步IO

区别：

  • 同步IO（Sync IO）：每进行一次IO操作，都要等到此次操作结束才能执行其他操作
  • 异步IO（AIO）：用户可以在发出一个IO请求后立即发出另一个IO请求，当全部IO请求发送完毕后，等待所有IO操作的完成

优点：

  • IO Merge：将多个IO操作合并为一个IO，例如同步IO分3次读取3个连续的页，AIO中会判断出这3个页是连续的，AIO底层会发送一个IO请求一次读取3个页

注意：Native AIO需要操作系统提供支持，Windows和Linux都支持，Mac OSX不支持

在Innodb存储引擎中，read ahead方式的读取都是通过AIO，脏页的刷新（磁盘的写入）也都是AIO完成

刷新邻接页

工作原理：当刷新一个脏页时，存储引擎会检查该页所在区的所有页，如果发现脏页就一起刷新

优点：将多个IO写入操作合并为一个IO操作

问题：

  1. 可能把不怎么脏的页进行了写入，很快又变脏了（所以可控制是否使用该功能）
  2. 固态硬盘读写速度本身就很快了，没必要（传统机械硬盘建议开启该功能，读写性能优秀的固态硬盘可关闭）

Innodb启动、关闭与恢复

关闭：参数innodb_fast_shutdown

参数值 Innodb存储引擎行为
0 innodb需要完成所有的full purge和merge insert buffer，并将所有脏页刷新回磁盘（需要时间较长，innodb升级时必须调成0）
1（默认值） 不需要完成full purge和merge insert buffer，但要将缓冲池中的数据脏页刷新回磁盘
2 不进行上述的操作，而是将日志写入日志文件，保证不会丢失事务，在数据库下次启动时，进行恢复操作

恢复：参数innodb_forcr_recovery（大的数字包含前面的所有影响）

参数值 Innodb存储引擎行为
0 需要恢复时，进行所有的恢复操作；不能有效恢复时，将错误写入错误日志（用户可进行DML操作，其余参数之下不允许）
1 忽略检查到的corrupt页
2 阻止master thread的运行
3 不进行事务的回滚操作
4 不进行插入缓存的合并操作
5 不查看撤销日志（undo log），将未提交事务视为已提交
6 不进行前滚操作

Innodb存储引擎文件

表空间文件

  • Innodb存储引擎将所有存储的数据存放在表空间中，表空间是Innodb存储引擎逻辑结构的最高层
  • 用户可以指定多个文件组合的共享表空间（innodb_data_file_path），也可以为每个表指定独立表空间（innodb_file_per_table）
  • 单独的表空间仅存储该表的数据、索引和插入缓冲bitmap等信息，其余信息还是存放在默认的共享表空间中（所以共享表空间的大小依然会不断增大）

如果对事务进行回滚，innodb不会去收缩共享表空间的大小，但是会自动判断这些信息是否还需要，如果不需要，会将这些空间标记为可用空间供下次使用

重做日志文件

  • 作用：当实例或介质失败时，例如数据库所在主机掉电导致实例失败时，innodb会使用重做日志恢复倒掉电前的时刻，以此保证数据完整性

  • 工作原理：每个innodb存储引擎都至少有一个重做日志文件组，每个文件组下面至少有两个重做日志文件（用户也可以设置多个镜像日志组，讲不同的文件组放在不同的磁盘上），日志组中每个重做日志文件大小一致，以循环写入的方式运行。

  • 参数：

    参数 默认值 描述
    innodb_log_file_size 每个重做文件的大小（太大会导致恢复很慢，太小会导致多次切换，同时会频繁触发checkpoint导致性能抖动）
    innodb_log_files_in_group 2 日志文件组中重做日志文件的数量
    innodb_mirrored_log_groups 1（Innodb没有镜像） 日志镜像文件组的数量（提高重做日志的可用性）
    innodb_log_group_home_dir ./ 日志文件组所在路径

  • 重做日志条目结构：

    redo_log_type space page_no redo_log_body
    1字节，表示重做日志类型 可能小于4字节，表示表空间的ID，采用压缩的方式 表示页的偏移量，同样采用压缩的方式 表示每个重做日志的数据部分，恢复时调用相应的函数进行解析

  • 写入过程：从缓冲往磁盘写入时，是按写入的最小单位一个扇区进行写入，可以保证写入必定是成功的（所以重做日志的写入过程中不存在doublewrite）

  • 写入磁盘的条件

    • 主线程每秒刷新
    • 参数innodb_flush_log_at_trx_commit，表示在事务提交时处理重做日志的方式
    参数值 描述
    0 提交事务时不将重做日志缓存写入磁盘，而是等待主线程每秒刷新（可能丢失部分事务）
    1默认 在事务提交时将重做日志缓冲同步写入磁盘，即伴有fsync的调用（保证事务ACID中的持久性，保证每当事务提交时已经写入重做日志文件，fsync耗时较长）
    2 将重做日志异步写入磁盘，即写入到文件系统的缓存中（可能丢失部分事务，如果数据库挂了但是操作系统和服务器没挂，文件系统的缓存可能还在）

重做日志文件与二进制日志文件的区别：

  1. 记录的范围不同：二进制日志会记录与Mysql数据库有关的日志记录，包括innodb、myisqm、heap等其他存储引擎的日志；而重做日志文件只记录有关innodb存储引擎本身的日志
  2. 记录的内容不同：二进制日志文件记录的是一个事务的具体操作内容，是逻辑日志；而重做日志记录的是每个页的更改的物理情况
  3. 写入时间不同：二进制日志文件仅在事务提交前写入，只写磁盘一次；而在事务进行的过程中，重做日志条目是不断地被写入重做日志文件中的

04-表

索引组织表

描述：在Innodb存储引擎中，表都是根据主键顺序组织存放的，这种存储方式称作索引组织表（数据即索引，索引即数据）

主键：

  • 每张表都有主键
  • 主键的定义或选择：
    • 建表时显式定义
    • 未显式定义，则存储引擎判断表中是否有非空的唯一索引，若有，该列即为主键（选择建表时第一个定义的非空唯一索引）
    • 若以上都不符合，则存储引擎自动创建一个6字节的指针

Innodb逻辑存储结构

表空间

见表空间文件

段

描述：表空间的组成部分，常见的段有数据段、索引段、回滚段等

数据段：B+树的叶子节点（索引节点）

索引段：B+树的非叶子节点（非索引节点）

工作原理：每个段开始时，先申请32个页大小的碎片页来存放数据，在使用完这些页之后才是64个连续页的申请（创建的表默认大小是96kb，在开始时申请较少的空间，节省磁盘容量开销）

区

描述：区是由物理上连续的页组成，区的大小固定是1mb（页的大小变化只影响区中页的数量，不改变区的大小）

页

描述：磁盘管理的最小单位，默认大小16kb（1.2版本后可以通过innodb_page_size设置大小4k、8k、16k，不可修改）

常见类型：

  • 数据页
  • undo页
  • 系统页
  • 事务数据页
  • 插入缓冲位图页
  • 插入缓冲空闲列表页
  • 未压缩的二进制大对象页
  • 压缩的二进制大对象页

行

innodb存储引擎是面向行的，也就是说数据是按行存放的

每个页最多允许存放7992行数据（16kb / 2 - 200）

InnoDB行记录格式

1.0版本之前格式分类：

  • Redundant：为兼容之前的版本而保留
  • Compact

Compact行记录格式

设计目标：高效的存储数据，一个页中的数据存放越多，其性能越高

行溢出数据

Innodb存储引擎可以将一条记录中的某些数据存储在真正的数据页面之外（一般认为大对象列数据会有溢出数据，但也不一定，varchar类型同样有可能有行溢出数据）

问题：一个页的大小一般是16k，也就是16384个字节，但一个varchar类型的列最多就能存储65535个字节（所有varchar类型列的长度总和），这时一个页可能存不下一条记录，就出现了行溢出

why？：Innodb存储引擎是索引组织的，即B+树结构，这样每个页中至少都应该有两条记录（否则B+树就失去了意义，变成链表了），因此，如果页中只能放下一条记录，那么Innodb存储引擎会自动将行数据放在溢出页中

解决：对于占用存储空间非常大的列，在记录的真实数据出只会存储该列的一部分数据，把剩余的数据分散存储在几个其他的页中进行分页存储，然后记录的真实数据处用20个字节存储指向这些页的地址

在大多数情况下BLOB类型的数据还是会发生行溢出，实际数据保存在BLOB页中，数据页只保存数据的前768字节

Compressed和Dynamic行记录格式

改变：这两种格式对存放在BLOB中的数据采用了完全行溢出的方式，在数据页中只存放20个字节的指针，实际数据都在Off Page中

Compressed行记录格式还有一个功能，对存储在其中的行数据以zlib算法进行压缩，对varchar、blob、text类型的大数据能够非常有效的存储

Char的行结构存储

约束

关系型数据库和文件系统的区别：

  • 关系型数据库本身就能保证存储数据的完整性，不需要应用程序的控制
  • 文件系统一般需要在程序段进行控制

约束机制保证数据库中的数据完整性，完整性有以下三种形式：

  • 实体完整性保证表中有一个主键：在Innodb存储引擎表中，用户可以通过定义Primary Key或Unique Key约束来保证实体完整性，还可以通过编写触发器来保证数据完整性
  • 域完整性保证数据每列的值满足特定的条件：可以选择合适的数据类型确保数据值满足特定条件，也可以添加外键约束、编写触发器，也可以考虑用DEFAULT约束作为强制域完整性的一个方面
  • 参照完整性保证两张表之间的关系：Innodb支持外键，因此允许用户定义外键以强制参照完整性，也可以编写触发器以强制执行

约束分类：

  • Primary Key：主键约束
  • Unique Key：唯一约束
  • Foreign Key：外键约束
  • Default
  • NOT NULL

约束创建方式：

  • 表建立时就进行约束定义
  • 利用ALTER TABLE命令来建立约束

约束和索引的区别：

用户创建了一个唯一的索引就创建了一个唯一的约束，但是约束是一个逻辑的概念，用来保证数据完整性；索引是一个数据结构， 不仅有逻辑上的概念，在数据库中还代表着物理存储的方式

错误数据:

  • 数据库本身没有对数据的正确性进行约束，可以插入或更新一个不正确的值（或者在内部将其转化成了一个合法值）
  • sql_mode：严格审核输入的参数，可以用来约束非法数据的插入或更新

触发器：

  • 作用：在执行INSERT、DELETE、UPDATE命令之前或之后自动调用sql命令或存储过程，可以实现一些mysql数据库本身不支持的一些特性（Check约束、物化视图、高级复制、审计等）
  • 特点：
    • 只有具有Super权限的用户可以执行Create Trigger命令来创建触发器
    • 一张表最多有六个触发器，对应前面提到的三种命令的Before和After
    • mysql数据库只支持按每行记录进行触发

外键约束：

  • 创建：create table时添加或表创建后通过alter table命令来添加
  • 被引用的表为父表，引用的表为子表，外键定义时ON DELETE和ON UPDATE表示对父表进行delete和update操作时对子表的操作，可定义的子表操作：
    • CASCADE：父表delete和update时，子表对应数据也进行delete和update操作
    • SET NULL：父表delete和update时，子表对应数据更新为NULL（子表对应列必须允许NULL值）
    • NO ACTION：父表delete和update时，若有子表关联此数据，抛出错误，不允许此类操作
    • RESTRICT：（默认外键设置）父表delete和update时，若有子表关联此数据，抛出错误，不允许此类操作
  • 特点：
    • 延时检查：其他数据库，如Oracle的外键约束是在sql语句运行完成后进行检查
    • 即时检查：mysql的外键约束都是即时检查，所以在mysql中NO ACTION和RESTRICT功能一样
    • Oracle建立外键的列需要手动为这个列添加一个索引，但mysql会自动为该列添加索引，可以避免外键列上五索引而导致的死锁问题

视图

视图：由查询定义的虚拟表

物化视图：Oracle数据库支持，该视图不是基于表的虚表，而是根据表示即存在的实表

mysql本身是不支持物化视图的，可以通过编写触发器的方式实现类似物化视图的功能，因此对于物化视图的查询重写功能等无能为力

分区表

分区不是存储引擎层完成的事，Innodb、MyISAM、NDB都支持，也有些CSV、Fedorated等不支持

分区：分区的过程就是将一个表或索引划分成多个更小、更可管理的部分

分区种类：

  • 水平分区：将同一表中不同行的记录分配到不同的物理文件中
  • 垂直分区：将同一表的不同列的记录分配到不同的物理文件中
  • 局部分区：一个分区中既存放了数据又存放了索引
  • 全局分区：数据存放在各个分区中，但数据的索引存放在一个对象中

特点：

  • 如果表中存在主键或唯一索引，分区列必须是唯一索引的一个组成部分
  • 如果坚表示没有指定主键或唯一索引，可以指定任一列为分区列

Mysql支持水平分区、局部分区索引，mysql支持分区类型：

  • RANGE分区：行数据基于一个给定连续区间的列值被放入分区
  • LIST分区：和RANGE分区类似，只不过面向的是离散的值
  • HASH分区：根据用户自定义的表达式的返回值（不能是负数）来进行分区，数据均匀地分布在预先定义的各个分区
  • KEY分区：与HASH分区类似，根据Mysql提供的哈希函数来进行分区
  • COLUMNS分区：可以使用非整型的数据进行分区，分区根据类型直接比较而得

子分区：

  • 在分区的基础上再分区，也叫复合分区，mysql允许在range和list的分区上在进行hash或key的子分区
  • 每个子分区的数量必须相同
  • 要在一个分区表的任何分区上使用SUBPARTITION来明确定义任何子分区，就必须定义所有的子分区
  • 每个SUBPATITION子句必须包括子分区的一个名字
  • 子分区的名字必须唯一

分区中的NULL值：

  • Mysql允许对NULL值进行分区，将NULL视作小于任何一个非NULL值
  • 不同分区的处理方式不同：range如果插入null值，放入最左边的分区；list必须显示指明要将null值放入那个分区，否则报错；hash和key分区任何的分区函数都会将含有null值的记录返回为0

分区和性能：

  • 分区不一定能提升查询性能，反而可能起反效果！一定要根据数据库使用环境合理使用分区
  • 数据库应用类型：
    • OLTP在线事务处理：通常不需要返回一张大表中10% 的数据，大部分只需要通过索引返回几条记录即可，仅B+树索引靠2~3次磁盘IO即可，分区反而可能带来严重的性能问题，必须要小心
    • OLAP在线分析处理：大多数的查询需要频繁扫描一张很大的表，分区可以很好的提高性能

05-索引

B+树

数据结构：

  • 二分查找
  • 二叉搜索树
  • 平衡搜索树/红黑树：自平衡，左右子树高度差不超过1
  • B树：多叉树
  • B+树：非叶子节点只存放索引，叶子节点存放数据，叶子节点之间构成一个有序链表

B+树是为磁盘或者其他直接存取辅助设备设计的一种平衡查找树，所有记录节点都是按照键值的大小顺序放在同一层的叶子节点上，有个叶子节点指针进行连接

插入：

  • 不论如何变化，B+树总是会保持平衡，但是对于新插入的值可能需要做大量的拆分页的操作，B+树的结构主要应用于磁盘，页的拆分意味着磁盘操作，应该尽可能减少页的拆分，故B+树页有类似于平衡二叉树的旋转功能：在Leaf Page已满，但左右兄弟节点没有满的情况下，首先将记录移到所在页的兄弟节点上，通常先检查左兄弟

删除：

  • 填充因子最小为50%

B+树索引

B+树索引的本质就是B+树在数据库的实现

B+索引在数据库的一个特点是高扇出性（说人话就是树越高磁盘IO次数也越多，IO次数越少越好），因此数据库中B+树的高度一般在2~4层

聚集索引

  • 聚集索引就是按每张表的主键构造一棵B+树，叶子节点（也叫数据页）中存放的记为整张表的行记录数据，非数据页存放的仅仅是键值以及指向数据页的偏移量
  • 每张表只能有一个聚集索引
  • 聚集索引的存储并不是物理上连续的，而是逻辑上连续的：页和页中的记录都是通过双向链表连接的，页按照主键顺序排序，页的记录可以不用按照主键存储

好处：

  • 聚集索引能特别快的访问针对范围值的查询
  • 对于主键的排序查找和范围查找特别快，叶子节点的数据就是用户想要查找的数据

辅助索引（非聚集索引）

  • 辅助索引的叶子节点并不包含行记录的全部数据，只记录了主键
  • 书签：叶子节点除了包含键值以外，每个叶子节点的索引行中还包含了一个书签，书签用来告诉Innodb存储引擎哪里可以找到与索引相对应的行数据
    • 在Innodb存储引擎中，书签就是行数据的聚集索引键（就是主键）
    • 对于其他数据库，有一些表类型为堆表，数据的存储按插入的顺序，堆表的特性决定了其上的索引都是非聚集的，主键与非主键的区别只是是否唯一且非空，这时的书签是一个行标识符。
  • 每张表可以有多个辅助索引
  • 查找过程（二级查找）：Innodb存储引擎会遍历辅助索引并通过叶节点的指针获得指向聚集索引的指针（主键），然后通过主键值在聚集索引来查找一个完整的行记录（这个过程也叫回表）

B+树索引的分裂

B+树索引页的分裂如果总是从页的中间记录开始的，会导致空间的浪费

PAGE_LAST_INSERT、PAGE_DIRECTION、PAGE_N_DIRECTION告诉Innodb存储引擎到底是向左还是向右分裂，同时分裂点记录为哪一个

索引使用

联合索引：

  • 说明：对表上的多个列进行索引
  • 联合索引也是一棵B+树，与单个键值的B+树一样，键值都是排序的，不同的是键值数量不是1，而是大于等于2
  • 最左前缀原则：如果创建（a,b,c）联合索引，查询时可以查a、a+b、a+b+c，其余情况无法匹配导致索引失效，（a+c属于索引截断，比较特殊，不同版本处理不一样）
  • 建立联合索引的时候，一定要把最常用的列放在最左边

覆盖索引：

  • 说明：从辅助索引中就可以得到的查询记录，而不需要查询聚集索引中的记录（说人话就是如果查找主键，在辅助索引里面就查到了，不用二级查找）
  • 好处：辅助索引不包含整行记录的索引信息，故其大小远远小于聚集索引，大量减少了io操作

哈希算法

Innodb存储引擎使用哈希算法对字典进行查找，采用链表方式解决哈希冲突，采用除法散列作为哈希函数

链接法：把散列到同一槽的所有元素放到一个链表里面

除法散列：通过取k除以m的余数，将关键字k映射到m个槽的某一个中去

数据库中m的取值：略大于缓冲池中页数2倍的质数。假设缓冲池大小设置为10m，则共有640个16kb的页，取略大于1280的质数1399，所以启动时分配1399个槽的哈希表用来哈希查询所有缓冲池中的页。

自适应哈希索引：

特点：

  • 自适应哈希索引由数据库自身创建，无法干预
  • 对于字典类型的查找很快，对范围查找无能为力

全文检索

全文检索：是将存储与数据库中的全书或全文中的任意内容信息查找出来的技术

06-锁

latch：轻量级的锁，锁定时间非常短，分为mutex（互斥量）和rwlock（读写锁），目的是为了保证并发线程操作临界资源的正确性，一般没有死锁检测的机制

lock：

  • 锁的对象是事务，用来锁定的是数据库中的对象，页表行
  • 锁住的对象仅在事务提交或回滚后释放（不同事务隔离级别施放时间不同）
  • 有死锁机制

行级锁：

  • 共享锁，允许事务读一行数据（读锁）
  • 排他锁，允许事务删除或更新一行数据

意向锁：将锁定对象分为多个层次，意向锁意味着事务希望在更细的粒度上加锁。Innodb支持意向锁设计的比较简单，其意向锁即表级别的锁，目的主要是为了在下一个事务中揭示下一行将被请求的锁类型（意向锁的目的是为了快速判断表里是否有记录被加锁，并不会真的锁住表）

  • 意向共享锁，事务想要获取一张表中某几行的共享锁
  • 意向排他锁，事务想要获取一张表中某几行的排他锁

排他锁实际上的意义是一个事务对记录加排他锁后其他事务不能再加任何锁，但是在InnoDB存储引擎中，简单的select……from……查询是不涉及锁机制的，也就是以下的一致性非锁定读，运用到了快照机制，所以加了排他锁的记录不影响简单查询，但MyISAM存储引擎的select操作会自动加读锁，所以是无法读到有排他锁的记录的

一致性非锁定读

  • InnoDB存储引擎通过多版本控制的方法来读取当前执行时间数据库中行的数据
  • 如果读取的行被加过排他锁，Innodb不会去等待锁的释放，而是去读取行的一个快照数据
  • 好处：快照是之前版本中的数据，通过undo段实现的，不需要额外的开销，快照数据也无需上锁，非锁定读机制大大提高了数据库的并发行
  • InnoDB默认的读取方式是非锁定读，但不同事务隔离下读取的方式不同，对快照的定义也不同：
    • Read Committed下，总是读取最新的一份快照数据
    • Repeatable Read下，总是读取事务开始时的行数据版本

自增长和锁：

  • AUTO-INC Locking：根据自增长计数器值加1赋予自增长列，是一种特殊的表锁机制，为了提高插入性能在完成对自增长插入的sql语句后立即释放（自增长列必须是索引，同时必须是索引的第一个列）
  • 缺点：并发插入性能差，必须等前面的插入完成；大数据量的插入会影响插入性能，阻塞其他事务的插入
  • 5.1后改进：
  • innodb_autoinc_lock_mode：控制自增长模式

外键和锁：

  • 对于外键值的插入或更新，首先需要查询父表中的记录，即SELECT父表
  • 但是对于父表的SELECT操作，不是使用一致性非锁定读的方式，因为这样会发生数据不一致的问题，因此这时使用的是SELECT…LOCK IN SHARE MODE方式，即主动对父表加一个S锁。如果这时父表上已经有X锁，子表上的操作会被阻塞

锁的算法

行锁算法：

  • Record Lock：单个行记录上的锁
  • Gap Lock：间隙锁，锁定一个范围，但不包含记录本身
  • Next-Key Lock：前面两个综合，锁定一个范围，并锁定记录本身

幻读

  • Phantom Problem：是指在同一事务下，连续执行两次同样的sql语句可能导致不同的结果，第二次的sql语句可能会返回之前不存在的行（而Innodb默认的隔离级别是可重复读，也就是在一个事务中，两次执行同样的查询，得到的效果应该是一样的，因此若事务b在事务a未提交时添加了一条记录，为了维护可重复读，事务a应该都是查询不到新增的记录的，只不过对于真实的表而言数据确实是增加了）
  • 另一种说法：事务 A 根据条件查询得到了 N 条数据，但此时事务 B 删除或者增加了 M 条符合事务 A 查询条件的数据，这样当事务 A 再次进行查询的时候真实的数据集已经发生了变化，但是A却查询不出来这种变化，因此产生了幻读。

幻读 VS 不可重复读：

  • 幻读：重点在于数据是否存在。原本不存在的数据却真实的存在了，这便是幻读。在同一个事务中，第一次读取到结果集和第二次读取到的结果集不同。（对比上面的例子，当B事务INSERT以后，A事务中再进行插入，此次插入相当于一次隐式查询）。引起幻读的原因在于另一个事务进行了INSERT操作。
  • 不可重复读：
    • 重点在于数据是否被改变了。在一个事务中对同一条记录进行查询，第一次读取到的数据和第二次读取到的数据不一致，这便是不可重复读。引起不可重复读的原因在于另一个事务进行了UPDATE或者是DELETE操作
    • 一般情况下，不可重复读的问题影响不大，因为读到的本来就是已经提交的数据
    • Oracle、Microsoft SQL Server默认隔离级别就是不可重复读
    • 但在Mysql中Innodb将不可重复读的问题定义为幻读，默认的隔离级别是可重复读

快照读 VS 当前读：

  • 快照读：读取的是快照数据，不加锁的简单select查询都属于快照读
  • 当前读：读取的是最新数据，而不是之前版本的快照数据，加锁的select或者数据的增删改都属于当前读

解决幻读：

  • 首先普通的行锁Record Lock是无法解决幻读问题的，行锁只能阻止修改，无法组织数据的删除，同时对于新插入的数据，原本就是数据库中不存在的数据，不可能加锁
  • 因此引入了新的锁Gap Lock，锁住的是两个值之间的间隙，这个间隙之中其他事务是无法插入数据的，Innodb采用的是行锁+间隙锁的Next-Key Lock方法
  • 特点：
    • 间隙锁再可重复读隔离级别才会生效
    • 间隙锁之间是不会产生冲突的

锁问题

脏页 VS 脏读

  • 脏页
    • 在缓冲池中已被修改的页，但还没有刷回磁盘
    • 数据库实例内存和磁盘异步造成的，不影响数据一致性，不影响数据库可用性，可以带来性能的提高
  • 脏读
    • 脏数据：事务对缓冲池中的记录的修改，但还未提交
    • 如果读到了脏数据，也就是另一个事务还没有提交的数据，显然违背了事务的隔离性
    • 仅发生在Read Uncommitted隔离级别下

丢失更新

  • 原因：一个事务的数据更新操作被另一个事务的更新操作覆盖，从而导致数据不一致
  • 一般数据库锁的存在不会发生理论上更新丢失的问题，另一个事务对同一记录进行修改要等到当前事务提交释放锁才行
  • 在应用的逻辑层面的丢失更新问题会比较严重，比如这种情况，User1的修改就会被覆盖：
    1. 事务T1查询一行数据，放人本地内存，并显示给一个终端用户User1
    2. 事务T2也查询该行数据，并将取得的数据显示给终端用户User2
    3. User1修改这行记录，更新数据库并提交
    4. User2修改这行记录，更新数据库并提交。
  • 解决：把事务串行化，不要并行操作，在查询时给读取的记录加上排他锁，那么步骤2必须等到1、3结束后才能执行，也就避免了逻辑上的更新丢失

阻塞

  • 一般情况下阻塞不是坏事，是可以确保事务可以并发且正常的运行
  • 问题：Innodb默认不会回滚超时引发的错误异常，如果一个事务一部分sql语句执行成功，后面的语句由于阻塞且等待超时导致执行失败，虽然此时事务可能会抛出异常，但是它并没有提交也不会回滚，这种状态非常危险，所以用户必须判断这个事务是提交还是回滚再进行下一步（保证事务的原子性）

死锁

  • 两个或两个以上的事务在执行过程中，因争夺锁资源导致的一种互相等待的现象
  • 粗暴解决办法：所有事务都不要等待，所有的等待都变成回滚，事务重新开始（降低并发性能，还会导致有可能所有事务都不能进行，牛）
  • 不那么粗暴：超时机制，两个事务相互等待，当其中一个事务等待时间超过阈值时回滚，另一个事务就可以进行下去了（如果回滚的事务操作量比较大，浪费时间）
  • Innodb的选择：等待图，前面都是被动死锁，这是主动检测死锁
  • 等待图：
    • 数据库保存两种信息：锁的信息链表、事务的等待链表
    • 通过两种链表可以构造出一张图，如果图中有回路，就代表存在死锁

锁升级

  • 将当前锁的粒度降低，行锁升级页锁，也锁升级表锁
  • 如果数据库设计时认为锁是稀缺资源，数据库就会频繁出现锁升级现象
  • 锁升级减少了维护锁的内存，但是可能降低并发效率

07-事务

事务特性ACID：

  • 原子性：事务是不可分割的工作单位，事务中任何一个sql语句执行失败，已经执行成功的sql语句也必须撤销，数据库的状态应该退回到执行事务前的状态
  • 一致性：数据库从一种状态转变为下一种一致的状态，事务是一致性的单位
  • 隔离性：事务的隔离性要求每个读写事务的对象对其他事物的操作对象能相互分离，即事务提交前对其他事务不可见
  • 持久性：事务一旦提交，其结果就是永久性的

事务分类

  • 扁平事务：最简单的一种事务类型，所有的操作都处于同一层次，是原子操作的最基本组成模块，同时它的限制也是不能提交或回滚事务的一部分或者分步提交

  • 带有保存点的扁平事务

    • 允许事务执行过程中回滚到同一事务较早的一个状态
    • 保存点：某些事物执行失败不代表所有操作都无效，保存点的作用是通知系统记住事务当前的状态，以便发生错误时事务回到保存点的状态即可（扁平事务有一个隐式的保存点放在事务开始）
    • 保存点是递增的，可以回滚到任意正确的保存点

  • 链事务：

    • 保存点模式的变种，保存点其实是易失的，一旦系统崩溃所有保存点都丢失了
    • 链事务：再提交一个事务时，释放不需要的数据对象，将必要的处理上下文隐式的传给下一个要开始的事务
    • 提交事务操作和开始下一个事务操作合并为一个原子操作，看上去就好像一个事务
    • 链事务的回滚仅限于当前事务，只能回滚到当前事务的起点

  • 嵌套事务：

    • 嵌套事务是由若干事务组成的一棵树，子树可以是嵌套事务也可以是扁平事务
    • 处在叶节点的是扁平事务
    • 根节点是顶层事务
    • 子事务既可以提交也可以回滚，但任何一个事务的回滚会引起它所有子事务一起回滚（子事务不具有持久性），所以任何事物只有顶层事务提交后才是真正意义上的提交了

    Innodb并不支持嵌套事务，可以通过保存点扁平事务模拟嵌套事务，但是嵌套查询父子事务之间的锁是可以继承或反继承的，模拟的方法没办法选择锁的继承或保留。而且如果系统支持子事务并行执行，模拟的方法根本无法满足，所以要实现事务之间的并行性，还得是嵌套事务

  • 分布式事务：

    • 通常是在分布式环境下运行的扁平事务，需要根据数据所在的位置访问网络中的不同节点
    • 比如跨行转账，涉及到网络中两个节点的数据库

log block

  • 重做日志块：重做日志缓存、重做日志文件都是以块的形式保存，一个块512字节（和磁盘扇区大小一样）

undo

  • 作用：
    • 如果事务因为某些原因执行失败，又或者用户用rollback请求回滚，是利用undo信息将数据回滚到修改之前的样子
    • Innodb中MVCC通过undo来完成，如果事务读取数据发现已经被其他事务占用，可通过undo读取之前版本的行信息，实现可重复读
    • undo log会产生redo log，所以undo log也需要持久性保护
  • undo段位于共享表空间
  • 注意：
    • undo并不是将数据库物理的恢复到执行事务的样子，只是逻辑上的恢复，数据库的结构和页本身可能大不相同，因为在多用户并发系统中，多个事务都在并发运行，如果一个事务回滚是完全恢复到执行事务之前的样子，可能对其他正在运行的事务产生影响
    • 回滚实际上是做与之前相反的工作，对每个insert完成一个delete，对delete执行一个insert，对update则执行一个相反的update
  • 存储管理：
    • 一堆参数，不想看了
    • undo log及其所在页在事务提交时先放入一个链表（其余事务可能还需要以前的版本），最终删除是由purge线程进行判断
    • innodb中undo页可以重用，节省空间
  • 格式：
    • insert undo log：insert操作产生的，事务提交就可以删除
    • update undo log：比insert的大

purge

  • 作用：最终完成delete和update操作
  • 原因：Innodb支持MVCC，所以记录不能在事务提交时立即处理，可能其余事务还要引用这些记录，所以innodb要保存以前的版本，当记录已不被其他事务引用，由purge进行判断清理之前的delete和update操作

group commit

  • 作用：提高磁盘fsync效率，一次fsync可以刷新确保多个日志被写入文件
  • 原理：事务提交时，日志先写入缓存，再刷新到磁盘，第二步相对第一步耗时较长，所以在有事务进行第二步时，将其他已完成第一步的事务一起刷盘
  • 开启二进制日志后，锁prepare_commit_mute以串行方式保证事物的顺序性，group commit失效，所以后面mysql 5.6采用了BLGC方式，大大提高了数据库的整体性
  • BLGC（Binary Log Group Commit）步骤：
    • Flush：将每个事务的二进制日志写入内存
    • Sync：将内存中的二进制日志刷新回磁盘，若队列中有多个事务，那么仅一次fsync操作就完成了二进制日志的写入
    • Commit：leader根据顺序调用存储引擎事务的提交（修复了由于锁prepare_commit_mutex导致group commit失效的问题）

事务控制语句

  • START TRANSACTION|BEGIN：显示开启一个事物
  • COMMIT：提交事务
  • ROLLBACK：回滚，结束当前事务，并撤销当前事务所有未提交的修改
  • SAVEPOINT identifier：创建一个保存点
  • RELEASE SAVEPOINT identifier：删除一个保存点
  • ROLLBACK TO[SAVEPOINT] identifier：回滚到一个保存点
  • SET TRANSACTION：设置隔离级别

事务操作统计:

  • QPS：每秒请求数
  • TPS：每秒事务处理能力，计算方法：（com_commit + com_rollback）/ time（所有提交都是显式的）

事务隔离级别

READ UNCOMMITTED：浏览访问

READ COMMITTED：游标稳定，大多数数据库默认

REPEATABLE READ：没有幻读保护（Innodb默认此隔离级别，使用next-key lock避免幻读）

SERIALIZABLE：SQL默认，Innodb在分布式事务时必须为此等级

分布式事务

  • 分布式事务：允许多个独立的事务资源参与到一个全局的事务中
  • 参与全局事务的所有事务要么都提交，要么都回滚，这对事务的ACID要求更高
  • 允许不同数据库之间的分布式事务
  • 事务组成：
    • 资源管理器：提供访问事务资源的方法，通常一个数据库就是一个资源管理器
    • 事务管理器：协调全局事务中的各个事务
    • 应用程序：定义事务的边界，指定全局事务中的操作
  • 两段式提交：
    1. 所有参与全局事务的节点都开始准备，告诉事务管理器准备好提交
    2. 事务管理器告诉资源管理器是提交还是回滚，任何一个节点不能提交则告知所有节点一起回滚
  • 内部分布式事务：存储引擎与插件之间、存储引擎与存储引擎之间可能还存在着分布式事务
